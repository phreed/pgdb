== Quick status for one service

[source,bash]
----
systemctl --user status <database>.service
----

Key fields to read:

`Active: active (running)`:: Service is up.
`Active: activating (start)`:: Still inside `HealthStartPeriod`.
`Active: failed`:: Crashed; see xref:#_read_logs[logs].

== Container health state

[source,bash]
----
podman ps --filter name=systemd-<database> \
          --format "{{.Names}}\t{{.Status}}"
----

Health states:

`(healthy)`:: Health check passing.
`(unhealthy)`:: Health check failing but container still running.
`(health: starting)`:: Inside the start period; checks not yet evaluated.

Run a one-off health check immediately:

[source,bash]
----
podman healthcheck run systemd-<database>
----

== Read logs

[source,bash]
----
journalctl --user -fu <database>.service
----

For the last 100 lines without following:

[source,bash]
----
journalctl --user -n 100 -u <database>.service
----

For multi-container setups, tail all at once (Aerospike example):

[source,bash]
----
journalctl --user -f \
  -u aerospike-server.service \
  -u aerospike-graph.service
----

== Diagnose a stuck startup

. Check the `HealthStartPeriod` — databases with slow JVM startup (Neo4j: 60 s, JanusGraph: 90 s, DataStax: 120 s) take longer before the first check runs.
. Check volume mounts are present: `podman volume ls | grep <database>`
. Check secrets exist: `podman secret ls`
. Check image was pulled: `podman image ls | grep <image-name>`
. Pull manually if needed: `podman pull <image>`

== Diagnose an unhealthy container

[source,bash]
----
podman inspect systemd-<database> \
  --format '{{json .State.Health}}' | jq .
----

This shows the last five health check outputs and exit codes, which reveals exactly what the health command printed when it failed.
