Podman quadlet definitions for 23 graph database engines, managed as rootless systemd user services on a Linux desktop or server.

This repository is part of the *Hydra* project, which builds pipelines that transform structured data (CSV, relational tables) into property graphs and RDF knowledge graphs.
The companion tutorial https://github.com/gdotv/tutorial-hydra-csv-to-graph[tutorial-hydra-csv-to-graph] demonstrates an end-to-end pipeline from CSV through GraphSON to RDF + SHACL, queried via Apache Jena Fuseki.
The quadlets defined here are the infrastructure layer that those pipelines connect to.

== Quick start

.Prerequisites
* https://podman.io[Podman] 4.4 or later
* https://pixi.sh[pixi] — manages Nushell and dotter as project-local tools
* https://github.com/AGWA/git-crypt[git-crypt] — for encrypting `podman.secret.toml` (optional but recommended)

.Install and start neo4j (the default enabled database)
[source,bash]
----
# Install symlinks and reload systemd
pixi run install

# Create Podman secrets from podman.secret.toml
pixi run create-secrets

# Start enabled services
pixi run start
----

See xref:_docs/tutorials/getting-started.adoc[Getting started] for a full walkthrough.

== Pixi tasks

[cols="<2,<5",options="header"]
|===
| Task | Description

| `pixi run install`
| Deploy quadlet symlinks with dotter and reload the systemd user daemon

| `pixi run create-secrets`
| Create Podman secrets from `podman.secret.toml` (add `-- --force` to overwrite)

| `pixi run start`
| Start all services listed in `.dotter/local.toml`

| `pixi run restart`
| Restart all failed or transitional services

| `pixi run status`
| Show systemd active state and Podman health for every quadlet service

| `pixi run status -- -w`
| Watch mode — refresh status every 5 s (or `-- -w -i 10` for 10 s)

| `pixi run trouble`
| Interactive troubleshooter: list failed services, inspect status / logs / health
|===

=== `systemd/`

One subdirectory per database, containing `.container`, `.volume`, and (for Aerospike Graph) `.network` quadlet files.
Dotter symlinks the selected databases into `~/.config/containers/systemd/pgdb/`.
Edit `.dotter/local.toml` to select which databases to deploy — only `neo4j` is enabled by default.

The 23 engines covered span property graph, RDF/triplestore, multi-model, and distributed backends.
See xref:_docs/reference/database-inventory.adoc[Database inventory] for the full list with images, ports, and protocols.

=== `podman.secret.toml`

Stores the values for all six Podman secrets (Neo4j auth, database passwords, RDFox license).
Encrypted with https://github.com/AGWA/git-crypt[git-crypt] before committing; ignored by git by default.
A sample with placeholder values lives at `_docs/sample-podman.secret.toml`.

First-time setup:

[source,bash]
----
git-crypt init
git-crypt add-gpg-user <your-gpg-key-id>
git add -f podman.secret.toml
----

== Relationship to tutorial-hydra-csv-to-graph

https://github.com/gdotv/tutorial-hydra-csv-to-graph[tutorial-hydra-csv-to-graph] walks through a complete data transformation:

. Read a veterinary clinic dataset from CSV files
. Convert to a property graph (GraphSON format)
. Encode the graph as RDF with SHACL constraints
. Load and query the RDF graph via *Apache Jena Fuseki*

This repository (`pgdb`) provides the running Jena Fuseki instance that the final step connects to, along with 22 other graph database engines for experimenting with alternative backends.
The `jena-fuseki` quadlet is enabled alongside `neo4j` when you want to run the full tutorial pipeline locally.

To add Jena Fuseki alongside the default Neo4j deployment, uncomment it in `.dotter/local.toml`:

[source,toml]
----
packages = [
    "jena-fuseki",
    "neo4j",
]
----

Then run:

[source,bash]
----
pixi run install && pixi run create-secrets && pixi run start
----
